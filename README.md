# E3RelaxH2: An Efficient High-Degree, High-Order Equivariant Graph Neural Network for Direct Crystal Structure Optimization


E3RelaxH2 is an E(3)-equivariant graph neural network that predicts relaxed crystal structures (atomic positions **and** lattice parameters) from unrelaxed inputs in a single forward pass.

## Preprocessing

### Download datasets (Zenodo)

The five datasets are available on Zenodo:

- https://zenodo.org/records/19212240

Archives:

- `c2db.tar.gz`
- `jarvis.tar.gz`
- `mp.tar.gz`
- `vdW.tar.gz`
- `xmno.tar.gz`

After uncompressing each archive, the dataset directory is expected to look like this:

```text
<dataset_root>/
├── CIF/          # all .cif files
├── train.csv
├── val.csv
└── test.csv
```

Example:

```text
/home/ubuntu/data/xmno/
├── CIF/
├── train.csv
├── val.csv
└── test.csv
```

Convert raw CIF pairs into LMDB shards that the training script expects.  
A single unified script handles all supported datasets:

```bash
python -m preprocess.preprocess \
    --dataset xmno \
    --data_root /your/real/path/to/dataset \
    --num_workers 4
```

> Replace `/your/real/path/to/dataset` with your own dataset root on your machine.  
> Example: `/home/ubuntu/data/xmno`.
> Also make sure `--dataset` matches the dataset in `--data_root` (or the dataset you intend to preprocess).

### Supported Datasets

| `--dataset` | Split source | ID column / filter | CIF naming |
|---|---|---|---|
| `xmno` | `id_prop_{split}_all.csv` | CSV col 0 filtered for `*_relaxed` | `{id}.cif` / `{id->unrelaxed}.cif` |
| `mp` | `{split}.csv` | `mp_id` | `CIF/{id}_R.cif` / `CIF/{id}_U.cif` |
| `c2db` | `{split}.csv` | `atoms_id` | `CIF/{id}_relaxed.cif` / `CIF/{id}_unrelaxed.cif` |
| `vdW` | `{split}.csv` | `atoms_id` | *(same as c2db)* |
| `jarvis` | `{split}.csv` | `atoms_id` | *(same as c2db)* |

Output directories are currently written to `{data_root}/train/`, `{data_root}/val/`, `{data_root}/test/`, each containing `data.0000.lmdb`, `data.0001.lmdb`, ...

## Training

Run from the project root (where `pyproject.toml` lives):

```bash
python -m train.train_E3RelaxH2 \
    --data_root /your/real/path/to/dataset \
    --dataset xmno \
    --batch_size 32 \
    --epochs 800 \
    --steps_per_epoch 800 \
    --early_stop_epoch 10 \
    --save_model
```

> Replace `/your/real/path/to/dataset` with your own dataset root on your machine.  
> Example: `/home/ubuntu/data/xmno`.
> `--dataset` should match the dataset in `--data_root` (for example, use `--dataset xmno` for XMNO data).
> If you use the output generated by `preprocess.preprocess` in this repo, point `--data_root` to `/your/real/path/to/dataset` so that `train/` and `val/` are found.

### CLI Arguments

| Argument | Default | Description |
|---|---|---|
| `--data_root` | *(required)* | Root directory containing `train/` and `val/` LMDB subdirs |
| `--dataset` | `xmno` | Dataset name: `xmno`, `mp`, `c2db`, `vdW`, `jarvis` |
| `--batch_size` | `32` | Training batch size |
| `--epochs` | `800` | Maximum number of pseudo-epochs |
| `--steps_per_epoch` | `800` | Training steps per pseudo-epoch |
| `--early_stop_epoch` | `10` | Patience (pseudo-epochs without improvement) |
| `--max_norm` | `50` | Gradient clipping max norm |
| `--num_workers` | `1` | DataLoader workers |
| `--save_model` | `False` | Save best checkpoint to wandb run dir |

### Is the trained model saved?

- If you pass `--save_model`, the script saves the **best** checkpoint as `model.pt` in the current wandb run directory. The saved model can typically be found at ./wandb/run-xxx/files/model.pt, where xxx represents specific run information.
- If you do **not** pass `--save_model`, no checkpoint file is saved.

Training uses:
- **AdamW** optimizer (lr=1e-4, no weight decay, AMSGrad)
- **ReduceLROnPlateau** scheduler (factor=0.8, patience=5)
- **Exponential Moving Average (EMA)** with 0.999 decay
- **wandb** for logging (project: `E3RelaxH2`)

## Evaluation

Use the saved checkpoint and your test LMDB split:

```bash
python -m test.test_E3RelaxH2 \
    --dataset xmno \
    --data_root /your/real/path/to/dataset \
    --model_path /your/real/path/to/model.pt \
    --test_subdir test \
    --batch_size 8
```

Notes:
- Replace all `/your/real/path/...` placeholders with your actual local paths.
- `--model_path` should usually be the `model.pt` path printed by training.
- If you use the output generated by `preprocess.preprocess` in this repo, point `--data_root` to `/your/real/path/to/dataset` so that the test split can be found.

This produces a CSV under `./results/` with per-structure metrics and also prints final average metrics at the end of evaluation.

| Column | Description |
|---|---|
| `cif_id` | Structure identifier |
| `mae_pos_Dummy` | Position MAE of unrelaxed vs. ground truth |
| `mae_pos_E3RelaxH2` | Position MAE of prediction vs. ground truth |
| `mae_metric_tensor_Dummy` | Metric tensor error (unrelaxed vs. ground truth) |
| `mae_metric_tensor_E3RelaxH2` | Metric tensor error (predicted vs. ground truth) |
| `mae_volume_Dummy` | Volume error (unrelaxed vs. ground truth) |
| `mae_volume_E3RelaxH2` | Volume error (predicted vs. ground truth) |
| `structure_match_*_gt` | pymatgen `StructureMatcher.fit()` result |
| `elapsed_time` | Inference wall time per batch |

To quickly inspect results (current helper uses a fixed CSV path in the script):

```bash
python -m test.check_result
```

## Model Architecture

E3RelaxH2 is built from the following components (in `model/block.py`):

1. **`MessagePassingBlock`** — E(3)-equivariant message passing with spherical-harmonic gating. Uses bilinear SH invariants and a learned reference direction per node.

2. **`SelfInteractionBlock`** — Gated self-interaction update that mixes scalar and vector features.

3. **`LatticeBlock`** / **`LatticeAxisBlock`** — Treats each lattice basis vector (a, b, c) as a virtual "axis node." Supports bidirectional message flow:
   - *Lattice → atoms*: injects lattice-aware features into atom representations.
   - *Atoms → lattice*: aggregates atom information to update lattice states.
   - *Axis–axis mixing*: couples the three basis vectors.

4. **`StructureUpdateBlock`** — Projects atom vector features to Cartesian position deltas.

Each layer iteratively:
1. Recomputes edge geometry from updated positions/cell
2. Lattice → atom interaction
3. Message passing between atoms
4. Self-interaction update
5. Position update
6. Atom → lattice aggregation + cell update

The model outputs a trajectory of `(pos, cell)` pairs — one per layer.
